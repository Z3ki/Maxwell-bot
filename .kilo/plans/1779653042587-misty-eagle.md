# Fix XML Tool Calling

## Goal
Keep XML as the tool-call format, but make it reliable. The current regex parser treats any `<tool:...>` anywhere in the assistant response as an executable call, including tool tags embedded inside `reasoning_log` attributes/bodies or JSON-like reasoning text. That caused shell commands and terminal actions to be corrupted or executed from the wrong layer.

## Findings
- `collect_tool_calls` in `bot.py` scans the whole response with a permissive regex and executes every available tool tag it finds.
- Production traces show the model often embeds `<tool:send_message>` or `<tool:shell>` inside `reasoning_log` text or JSON-looking reasoning payloads.
- The parser has no concept of top-level tool calls, so nested/leaked XML inside parameter text is treated as real calls.
- Attribute parsing is fragile for natural language because unescaped quotes, `<`, `>`, and multi-line values can break extraction.
- Shell normalization only rejects/recover-markup after the parser has already selected a shell call, so it cannot fix nested false positives.
- Tests cover happy-path XML but not malformed/nested production outputs.

## Plan
1. Define a stricter XML tool-call envelope while keeping XML.
   - Prefer top-level tool blocks only: one tool call per line or adjacent top-level tags.
   - Keep supported forms:
     - `<tool:react emoji="catjam" />`
     - `<tool:shell><command>neofetch</command></tool:shell>`
     - `<tool:send_message>hello</tool:send_message>`
   - Update prompt wording to strongly prefer sub-tags for any free text, command, HTML, code, or multi-line parameter.
   - Discourage putting tool tags inside `reasoning_log`; require reasoning fields to describe plans in plain text, not literal tool markup.

2. Replace regex-only scanning with a small top-level XML tokenizer/parser.
   - Iterate through the assistant response left-to-right.
   - Recognize only tags at top level, not tags found inside another tool body.
   - When parsing a non-self-closing tool, consume through its matching closing tag and do not scan its body for additional tool calls.
   - Parse child parameter tags only as parameters, never as executable tools.
   - Preserve mixed prose outside top-level tool calls so it can be cleaned or sent normally.

3. Add robust parameter parsing and unescaping.
   - Support quoted attributes for short scalar parameters.
   - Support sub-tags for long/free-text parameters.
   - Decode basic XML entities: `&amp;`, `&lt;`, `&gt;`, `&quot;`, `&apos;`.
   - For tool bodies without child parameter tags, map to existing default params (`send_message` -> `content`, `shell` -> `command`, `tts` -> `text`, etc.).
   - Do not parse nested `<tool:...>` inside a parameter value as a call.

4. Add malformed-output recovery without executing unsafe false positives.
   - If the whole response is a `reasoning_log` with embedded terminal tool tags inside its text, execute only `reasoning_log` and force a follow-up, rather than executing embedded terminal tags.
   - If a top-level `shell` call has no `command` parameter but contains obvious prose or leaked tool markup, return a tool error and ask for a follow-up, not execute markup.
   - If terminal tools (`send_message`, `no_response`) appear inside another tool body, treat them as text, not terminal decisions.

5. Fix terminal/follow-up behavior for shell.
   - Shell sends output directly to chat, so shell results should not require the model to invent a second visible answer unless there is an error or the model explicitly also called `send_message`.
   - Consider marking `__SHELL_SENT__` as terminal for `_tool_results_need_followup`, or require shell calls to end with `<tool:no_response />` and ensure nested no_response is not misparsed.
   - Prefer the safer approach: make `__SHELL_SENT__` terminal unless the result contains an error marker.

6. Update tests with production regressions.
   - Top-level XML calls still execute in order.
   - Nested `<tool:send_message>` inside `<tool:reasoning_log>...</tool:reasoning_log>` is not executed.
   - JSON-looking reasoning text containing XML tags is not parsed as tool calls unless wrapped as a top-level tool call.
   - `<tool:shell><command>neofetch</command></tool:shell>` executes with command `neofetch`.
   - `<tool:shell command="neofetch" />` still works.
   - Shell result does not trigger a follow-up when it succeeded.
   - `strip_tool_payload_leaks` removes only top-level tool payload leaks and does not aggressively delete normal XML/HTML/code in user-visible text.

7. Validate locally.
   - Run `pytest -q tests/test_tool_calls.py tests/test_tts_silent.py`.
   - Run broader tests if quick enough: `pytest -q`.
   - Restart `maxwell-bot` only after implementation and tests pass.

## Expected Outcome
The bot keeps XML tool calls, but only intentional top-level tool tags execute. Reasoning text can contain examples or leaked tags without triggering extra calls, shell commands stop being replaced by tool markup, and shell output stops causing unnecessary follow-up loops.
