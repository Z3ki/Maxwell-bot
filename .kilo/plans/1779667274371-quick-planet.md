# Plan: Fix VC Static Blast After Initial Hearing

## Problem
Maxwell now joins VC and captures speech, but logs show long runs of `Dropping corrupted voice packet ... corrupted stream` while the user is speaking. The current decoder patch catches `discord.opus.OpusError`, resets the decoder, then returns a 20ms silent PCM frame (`b"\x00" * 3840`). That prevents router crashes, but it still delivers synthetic frames to `LiveSpeechSink`.

Because `LiveSpeechSink` appends non-voiced frames while an utterance is active, these substitute frames can stretch and pollute the captured WAV after the first good audio. This matches the user symptom: Maxwell hears the start, then the captured audio turns into static/garbage and the model either responds incorrectly or returns `__NO_RESPONSE__`.

## Fix Strategy
Keep the router alive, but do not feed corrupted audio into utterance capture.

1. In `bot.py` `_patch_voice_recv_decoder()`:
   - Keep the DAVE/Opus protection wrapper.
   - On `discord.opus.OpusError`, reset `self._decoder` as already planned.
   - Return `(packet, b"")` instead of a fake silent PCM frame.
   - This makes `VoiceData.pcm` empty, so `LiveSpeechSink.write()` skips the packet via its existing `if not pcm: return` guard.
   - Preserve throttled warning logs so we can still see if corruption continues.

2. Improve DAVE handling in `bot.py`:
   - When DAVE decrypt fails with `UnencryptedWhenPassthroughDisabled`, call `session.set_passthrough_mode(True, 10)` if available.
   - Then mark that decoder/user as unencrypted passthrough and stop retrying DAVE for that stream, as current code already does.
   - Optionally check `session.can_passthrough(int(user_id))` when available before enabling passthrough, but do not block on it if it raises.
   - Goal: avoid repeatedly trying to DAVE-decrypt normal Opus packets while the session expects passthrough-disabled encrypted packets.

3. Harden `voice_live.py` against bad captures:
   - Track `bad_or_empty_frames` is not needed if corrupted frames are dropped before sink.
   - Add finalization quality gates using existing state:
     - Require a minimum voiced frame count, not just total duration.
     - Compute voiced duration from `voiced_frames * frame_dur` or store `voiced_seconds` in `_SpeakerState`.
     - Discard utterances with enough total duration but too little voiced audio.
   - Keep interruption behavior, but do not let a single loud corrupt/garbage frame interrupt unless it survives decode and exceeds threshold; this should be rare once Opus errors are dropped.

4. Reduce noisy but harmless logs:
   - Optionally lower `discord.ext.voice_recv.reader` RTCP `SenderReportPacket` logging to warning/error filtering only if it is drowning useful VC logs.
   - Do not change third-party package files; do any filtering in app logging setup if needed.

## Verification
1. Run `python -m py_compile bot.py voice_live.py`.
2. Restart `maxwell-bot` with PM2 after implementation.
3. Use `,vc join` and speak a phrase that previously caused the issue.
4. Inspect `pm2 logs maxwell-bot --lines 200 --nostream`:
   - No fresh `discord.ext.voice_recv.router - ERROR - Error in <PacketRouter...>` tracebacks.
   - Corrupted packet warnings may still appear, but captures should no longer include substitute silent/static frames.
   - `VC utterance captured` durations should match actual speech length more closely.
   - `VC utterance response` should occur for addressed speech; no response for non-addressed speech is acceptable.
5. If corruption remains severe after dropping frames:
   - Add temporary diagnostics for per-utterance dropped frame count and voiced duration.
   - Save one WAV for inspection instead of deleting it in `_handle_vc_utterance` only while debugging, then remove that diagnostic behavior.
