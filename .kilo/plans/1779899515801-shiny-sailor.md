# Plan: Simplify Tool Reasoning and Formatting Pipeline

## Diagnosis

The current setup is bad for the LLM because it mixes several competing formats:

- The prompt teaches XML-like tool calls: `<tool:reasoning_log><thoughts>...</thoughts></tool:reasoning_log>`.
- `ReasoningLogTool` asks for verbose metadata fields: `thoughts`, `intent`, `confidence`, `decision`, `assumptions`, `evidence`, `alternatives`, `risks`, `tool_plan`, `response_plan`, `data`.
- The model sometimes puts XML child tags inside the `thoughts` field itself, producing messy traces like `thoughts:<thoughts>...</thoughts><intent>...</intent>`.
- The parser accepts nested tags inside tool bodies as parameter text, so invalid-looking reasoning can still be recorded instead of rejected or normalized.
- The forced formatting prompt changes added earlier are too repetitive and may dilute the actual task.
- The regex auto-formatter is a brittle workaround and should be treated as a temporary fallback, not the main behavior.

## Goal

Make the bot easier for the LLM to follow by using one clear tool-call format, reducing reasoning-log verbosity, preventing nested XML inside `thoughts`, and keeping formatting instructions simple.

## Proposed Changes

1. Simplify `ReasoningLogTool.get_description()`

Replace the verbose metadata request with a compact schema:

- `thoughts`: one short plain-English sentence only
- `intent`: short label
- `decision`: short label
- `confidence`: optional `low`, `medium`, or `high`

Remove the instruction to include every possible field by default.

2. Add a hard rule against XML/JSON inside reasoning fields

In `_tool_system_prompt()`, add a clear rule:

- `reasoning_log` values must be plain text only.
- Never put `<thoughts>`, `<intent>`, JSON objects, or other tags inside a reasoning field.
- Use child tags only as the outer parameter syntax.

3. Prefer compact reasoning syntax examples

Change examples from verbose multi-field blocks to simple examples:

```xml
<tool:reasoning_log>
  <thoughts>User asked for a site, so I should create one.</thoughts>
  <intent>create_site</intent>
  <decision>use_create_site</decision>
  <confidence>high</confidence>
</tool:reasoning_log>
```

4. Sanitize `ReasoningLogTool` payloads before saving

Add a small sanitizer in `ReasoningLogTool.execute()`:

- Strip nested `<thoughts>...</thoughts>`, `<intent>...</intent>`, etc. from the `thoughts` field if the model accidentally includes them.
- If `thoughts` contains obvious serialized XML metadata, extract known fields into their proper keys.
- Clamp long fields to reasonable lengths for trace readability.

5. Reduce prompt conflict around formatting

Keep one formatting instruction in the system prompt, not three competing ones.

Recommended wording:

- Use Discord markdown when it naturally improves readability.
- Always format code, commands, file names, keys, links, and multi-item answers.
- Do not force markdown into tiny greetings or one-line jokes.

This avoids unrealistic rules like “Never send plain unformatted response,” which can make outputs weird.

6. Reconsider or remove regex auto-formatting

The current regex auto-formatter may create unnatural output by bolding arbitrary first clauses. Prefer relying on clearer prompt instructions first.

If keeping a fallback, restrict it to obvious technical output only:

- wrap product keys, commands, filenames, URLs, and code-ish tokens
- do not bold casual greetings or normal chat

7. Add tests

Add tests for:

- `reasoning_log` with nested XML inside `thoughts` gets normalized or stripped
- valid compact reasoning log is preserved
- formatting instructions are present once in `_build_messages()`
- casual text is not aggressively auto-bolded if the fallback remains

## Implementation Order

1. Update `ReasoningLogTool.get_description()` to compact plain-text metadata.
2. Update `_tool_system_prompt()` with a no-nested-tags rule and compact examples.
3. Add sanitization in `ReasoningLogTool.execute()`.
4. Remove duplicate/over-strong formatting instructions, leaving one clear instruction.
5. Remove or narrow `_auto_format_discord()`.
6. Run tests.
7. Restart `pm2 maxwell-bot` after validation.

## Expected Result

The LLM sees fewer competing instructions, produces cleaner tool calls, stops dumping XML inside `thoughts`, and formatting becomes less chaotic. The traces should become short, readable JSON records instead of nested XML-looking blobs.
